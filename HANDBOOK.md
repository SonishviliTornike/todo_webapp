დაწერილია შენი მენტორის მიერ. ორი ნაწილია:
**ნაწილი I** — პრინციპები, რომლებიც შენს პროექტზე უკვე ისწავლე.
**ნაწილი II** — სრული რუკა: ყველაფერი, რაც საიტს სჭირდება, არჩევანები
და მათი ფასი. რაც ჯერ არ გაგიკეთებია, მონიშნულია ⏳-ით — ეს შენი

# ნაწილი I — პრინციპები, რომლებიც უკვე ისწავლე

## 1. დიდი სურათი — რა ხდება ერთ request-ზე

ყოველი გვერდის ნახვა ერთი და იგივე გზას გადის:

ბრაუზერი → ვებ-სერვერი (nginx) → PHP → ერთი შესასვლელი წერტილი
(index.php) → router/dispatcher → controller → model (ბაზა) →
template → პასუხი ბრაუზერს.

რატომ ერთი შესასვლელი წერტილი: ყველაფერი, რაც ყველა გვერდს სჭირდება
(სესიის გახსნა, CSRF შემოწმება, login-ის შემოწმება), ერთ ადგილას კეთდება
და ვერცერთი გვერდი ვერ გამორჩება. ამას ჰქვია **Front Controller**.

გატყდება: თუ ყველა გვერდი ცალკე ფაილია, დაცვა ყველგან ცალ-ცალკე უნდა
ჩაწერო და ერთგან აუცილებლად დაგავიწყდება.

## 2. არქიტექტურა

**MVC — სამი როლი, სამი პასუხისმგებლობა.**
Model ბაზასთან ლაპარაკობს. Controller წყვეტს, რა მოხდეს. Template (View)
მხოლოდ აჩვენებს. არცერთი არ აკეთებს მეორის საქმეს.
გატყდება: SQL რომ template-ში ჩაწერო, ვერც შეამოწმებ, ვერც შეცვლი,
ვერც დაიცავ — ყველაფერი ერთ გორად იქცევა.

**ერთი კლასი = ერთი მიზეზი შესაცვლელად (Single Responsibility).**
თუ კლასს ორი სრულიად სხვადასხვა მიზეზით ცვლი, ორი კლასია ერთში ჩაჭედილი.

**Dependency Injection — კლასი თავის ხელსაწყოებს გარედან იღებს.**
კლასი შიგნით არ ქმნის PDO-ს ან სხვა კლასებს — constructor-ით იღებს.
რატომ: ასე კლასს ტესტში ყალბი (fake) დამოკიდებულებით შეამოწმებ და
ერთ დამოკიდებულებას მეორეთი ადვილად შეცვლი.
გატყდება: `new` კლასის შუაგულში = ვერასდროს გატესტავ იზოლირებულად.

**წესრიგი ფოლდერებში (PSR-4).**
Namespace = ფოლდერის გზა. კლასის სახელი = ფაილის სახელი. კლასი იმ
ფოლდერში უნდა იჯდეს, რა როლიც აქვს — routing კლასს Model ფოლდერში
ადგილი არ აქვს (შენი EntryPoint-ის გაკვეთილი).

**კოდის სტილი (PSR-12).**
ერთიანი სტილი მთელ პროექტში: ხაზის სიგრძე, ფრჩხილები, indentation.
რატომ: კოდს ბევრჯერ მეტს კითხულობენ, ვიდრე წერენ.

## 3. უსაფრთხოება — წესი ნომერი ერთი: არ ენდო შემოსულ მონაცემს

ყველაფერი, რაც მომხმარებლისგან მოდის (ფორმა, URL, cookie, header),
პოტენციურად მტრულია. ყველაფერი მოწმდება სერვერზე.

**SQL Injection — მონაცემი SQL-ში მხოლოდ placeholder-ით.**
Query-ს ტექსტი და მონაცემი ერთმანეთს არასოდეს ეწებება string-ად.
Prepared statements + bound values — ყოველთვის, გამონაკლისის გარეშე.
გატყდება: `' OR 1=1 --` და შენი ბაზა მისია.

**სვეტის სახელებს placeholder ვერ იცავს — მათთვის whitelist.**
Placeholder მხოლოდ მნიშვნელობებზე მუშაობს. თუ სვეტის სახელი დინამიურია,
ის მოწმდება ნებადართული სახელების სიაში. შემოწმება ერთ ადგილას ზის და
ყველა მეთოდი მასზე გადის.

**Mass Assignment — „ლეგალური" ჩანაწერი არალეგალურ სვეტში.**
შეტევა, სადაც SQL სწორია, უბრალოდ ისეთ სვეტს წერს, რომელსაც არ უნდა
წერდეს. შენი ბაზის მაგალითი: `user_role` სვეტი — თავდამსხმელი ფორმას
დაამატებდა `role=admin`-ს და ადმინად იქცეოდა. იგივე whitelist იცავს.

**XSS — გამოტანისას ყველაფერი escape-დება.**
რასაც ბაზიდან ან მომხმარებლისგან ეკრანზე აჩვენებ, გადის
`htmlspecialchars`-ში. რატომ: თუ ვინმე task-ის სათაურად `<script>`-ს
ჩაწერს, ის სხვის ბრაუზერში გაეშვება და სესიას მოიპარავს.
წესი: ვალიდაცია შესვლისას, escape გამოსვლისას. ორივე, არა ერთ-ერთი.

**CSRF — სხვა საიტიდან გაგზავნილი ყალბი ფორმა.**
ბრაუზერი cookie-ს ავტომატურად აყოლებს ყველა მოთხოვნას, ამიტომ მტრულმა
საიტმა შეიძლება შენი სახელით გააგზავნოს ფორმა. დაცვა: ყოველ
state-changing (POST) მოთხოვნას მოაქვს შემთხვევითი ტოკენი, რომელიც
სესიაში ინახება და ცენტრალურად მოწმდება. შედარება `hash_equals`-ით.

**ვალიდაცია whitelist-პრინციპით.**
არ ცდილობ „ცუდის" ამოცნობას — აღწერ, რა არის დასაშვები (ტიპი, სიგრძე,
ფორმატი, დასაშვები მნიშვნელობები) და ყველაფერ დანარჩენს აგდებ.

**ერთი წყარო სიმართლის (Single Source of Truth).**
ბაზის სვეტის სიგრძე და ვალიდაციის ლიმიტი ერთმანეთს უნდა ემთხვეოდეს.
შენი მაგალითი: ბაზაში VARCHAR(50), ვალიდაციაში 100 — 51-სიმბოლოიანი
სათაური = SQL შეცდომა = 500 გვერდი.

## 4. უსაფრთხოება — ვინაობა და სესია

**პაროლი მხოლოდ ჰეშად ინახება.**
`password_hash` + `password_verify`. ორიგინალი პაროლი არსად, არასოდეს —
არც ბაზაში, არც ლოგში. ჰეშირება ცალმხრივი გზაა.

**Session Fixation — და რატომ regenerate login-ზე.**
შეტევა: თავდამსხმელი მსხვერპლს საკუთარ session id-ს აჩეჩებს; მსხვერპლი
ამ id-თი შედის სისტემაში; თავდამსხმელს ავტორიზებული სესია რჩება ხელში.
დაცვა: წარმატებული login-ისას სისტემა id-ს ცვლის და ძველს შლის
(`session_regenerate_id(true)`). თანმიმდევრობა კრიტიკულია: ჯერ ახალი id,
მერე მომხმარებლის მონაცემების ჩაწერა სესიაში.

**სრული Logout — ორივე მხარე უნდა გასუფთავდეს.**
სერვერზე: სესიის მონაცემების წაშლა და სესიის განადგურება.
ბრაუზერში: cookie-ს ვადის გასვლა წარსული თარიღით.
წაშლის cookie-ს ზუსტად იგივე პარამეტრები (path, domain...) უნდა ჰქონდეს,
რაც ორიგინალს — თორემ ბრაუზერი მას სხვა cookie-დ ჩათვლის და ძველი დარჩება.

**Cookie-ს ატრიბუტები — თითოს თავისი საქმე აქვს.**
`HttpOnly` — JavaScript ვერ წაიკითხავს (XSS-ის შედეგს ამცირებს).
`SameSite` — სხვა საიტიდან მოსულ მოთხოვნას cookie არ მიჰყვება (CSRF-ს ეხმარება).
`Secure` — cookie მხოლოდ HTTPS-ით მოძრაობს (production-ში სავალდებულო).

**გახსოვდეს განსხვავება:** `$_COOKIE` = რაც ბრაუზერმა გამოგზავნა ამ
მოთხოვნით. `session_get_cookie_params()` = შენივე კონფიგურაცია, რომლითაც
PHP cookie-ს გასცემს. ბრაუზერი ატრიბუტებს (path, httponly) უკან არასოდეს
აბრუნებს — მხოლოდ სახელს და მნიშვნელობას.

**Ownership — ყველა ჩანაწერი თავის პატრონს.**
ყოველი წაკითხვა, შეცვლა, წაშლა შემოწმდება: ეკუთვნის თუ არა ჩანაწერი
ამ მომხმარებელს (`WHERE ... AND user_id = ?`). user_id ყოველთვის
სესიიდან მოდის, არასოდეს ფორმიდან.
გატყდება: URL-ში id-ს შეცვლით სხვისი მონაცემები იშლება/იკითხება.
ამ შეცდომას ჰქვია IDOR და რეალურ საიტებზე ყველაზე გავრცელებული ხვრელია.

**არ გაამხილო, რა არსებობს.**
სხვისი task → 404 („არ არსებობს"), არა 403 („არსებობს, მაგრამ არა შენია").
Forgot password → „თუ ეს email არსებობს, ბმული გავაგზავნეთ" — არასოდეს
თქვა, რომელი email-ია დარეგისტრირებული. ამას email enumeration ჰქვია.

**Brute force — ცდების შეზღუდვა.**
Login-ზე ცდების მთვლელი + შეყოვნება. უამისოდ პაროლის გამოცნობა
მხოლოდ დროის საკითხია.

**Reset-ტოკენების წესები (forgot password).**
ტოკენი: შემთხვევითი, ერთჯერადი, ვადიანი, ბაზაში მხოლოდ მისი **ჰეში**
ინახება (თუ ბაზა გაიტაცეს, ტოკენები უსარგებლოა). გამოყენებისას და
პაროლის შეცვლისას — უქმდება.

## 5. შეცდომების მართვა

**Exception-ების იერარქია იცოდე.**
PHP-ში `Error` (TypeError და მისთანები) და `Exception` ორივე `Throwable`-ის
შვილია, მაგრამ ერთმანეთის არა. `catch (Exception)` TypeError-ს **ვერ**
დაიჭერს. ბოლო ბადე ყოველთვის `catch (\Throwable)`.
`InvalidArgumentException` არის `LogicException`-ის შვილი და არა
`RuntimeException`-ის — შენივე პროექტში წამოგედო ეს.

**Fail Fast — გაჩერდი მაშინვე, გამოყენებამდე.**
შემოწმებები (guard clauses) მეთოდის თავშია, ერთად. ჯერ ვამოწმებთ, მერე
ვიყენებთ. არასწორი მონაცემით query-ს აშენებაც კი არ უნდა დაიწყოს.
გატყდება: შემოწმება რომ გამოყენების **მერე** დგას, refactoring-ისას
ვინმე რიგს შეცვლის და დაცვა ჩუმად მოკვდება.

**შეცდომა მომხმარებელს — ზოგადი, ლოგს — დეტალური.**
ეკრანზე: „სერვისი მიუწვდომელია". ლოგში: სრული exception, ფაილი, ხაზი,
კონკრეტული მნიშვნელობა, რამაც ჩააგდო. Production-ში `display_errors=Off`
სავალდებულოა — ეკრანზე გამოტანილი შეცდომა თავდამსხმელისთვის რუკაა.

**HTTP სტატუს კოდები სიმართლეს ამბობენ.**
200 = მოხერხდა, 302 = გადამისამართება, 400 = შენი მოთხოვნა მცდარია,
403 = აკრძალულია, 404 = არ არსებობს, 500 = ჩვენი შეცდომა,
503 = სერვისი დროებით ვერ პასუხობს. Redirect-ის წინ ხელით 200-ის დაყენება
აზრს მოკლებულია — redirect 302-ია.

## 6. კოდის ხარისხი

**DRY — ერთი ლოგიკა, ერთი ადგილი.**
ერთი და იგივე შემოწმება სამ ადგილას = სამი ადგილი გასაფუჭებლად.
ლოგიკა ერთ helper-ში ზის, ყველა მას იძახებს — წესი რომ შეიცვალოს,
ერთ ადგილას ასწორებ.

**Choke Point — დაცვა იქ, სადაც გვერდის ავლა შეუძლებელია.**
helper-ის გამოძახება კონვენციაა — ახალმა მეთოდმა შეიძლება დაივიწყოს.
უფრო ძლიერი დიზაინი: შემოწმება ჩაშენებულია იმ ერთადერთ გზაში, რომელსაც
ყველა აუცილებლად გაივლის (მაგ. query-ს აშენების ფუნქციაში). მაშინ
დავიწყება ფიზიკურად შეუძლებელია.

**თანმიმდევრულობა.**
ერთი და იგივე რამ ყველგან ერთნაირად კეთდება: შეცდომების ერთი ფორმატი,
ერთი სტილი, ერთი მიდგომა. როცა ერთი ადგილი განსხვავდება, მკითხველი
ფიქრობს „ეს განზრახ არის?" და დროს კარგავს.

**Typo იდენტიფიკატორში ბაგია და არა კოსმეტიკა.**
შენი სტატისტიკა: რვა typo ერთ დღეში, ერთმა მათგანმა მთელი whitelist
გააუქმა. წესი: commit-მდე კოდს ერთხელ ისე გადაუყევი, თითქოს სხვისია.

**წაკითხვადი API — Boolean Trap.**
`func(true, false, true)` — ვერავინ იტყვის, რას ნიშნავს. სახელიანი
პარამეტრები / options-მასივი თავად ხსნის თავს. ეს არის მიზეზი, რის გამოც
`setcookie`-ს მასივიანი ფორმა ჯობია პოზიციურს.

**მკვდარი კოდი იშლება.**
გამოუყენებელი import, ცარიელი constructor, სვეტი ბაზაში რომელსაც არავინ
კითხულობს — ან გამოიყენე, ან წაშალე. ნახევრად გაკეთებული ორივეზე უარესია.

## 7. ტესტირება — „მგონი მუშაობს" არ ითვლება

**ყოველ ცვლილებას ორი ტესტი: დადებითი და უარყოფითი.**
დადებითი: სწორი გზა მუშაობს. უარყოფითი: შეტევა/არასწორი მონაცემი
**ეტყობა უარით**. უარყოფით ტესტში წარმატება = სისტემამ სწორად თქვა უარი.
ვინც მხოლოდ happy path-ს ტესტავს, დაცვას არასოდეს ამოწმებს.

**ბრაუზერის ტესტი მხოლოდ იმ გზებს ფარავს, რომლებზეც გაიარე.**
კოდის გზა, რომელსაც UI-დან ვერ მიწვდები, ბრაუზერით ვერასოდეს შემოწმდება
(შენი DatabaseTable::update()-ის გაკვეთილი). ასეთ გზებს მხოლოდ
unit-ტესტი ფარავს.

**იცოდე, რომელი ხელსაწყო რას იჭერს.**
`php -l` — მხოლოდ სინტაქსს. Undefined variable-ს ვერ ხედავს.
PHPStan — undefined variable-ს, ტიპების შეცდომებს, მკვდარ კოდს, commit-მდე.
PHPUnit — ლოგიკის სისწორეს, მიუწვდომელი გზების ჩათვლით.
ბრაუზერი + ლოგები — რეალურ ქცევას.

**მტკიცებულება, არა შთაბეჭდილება.**
„მუშაობს" არ არის ანგარიში. ანგარიშია: ლოგის ხაზი, Network tab-ის header,
შეცვლილი session id-ის მნიშვნელობა. DevTools-ის Network და Application
tab-ები შენი მეგობრებია.

## 8. Git-დისციპლინა

**ერთი ლოგიკური ცვლილება = ერთი commit.**
რატომ: ასე ცვლილება ცალკე ბრუნდება უკან, ცალკე იკითხება, ცალკე მოწმდება.

**Commit-მდე ყოველთვის: `git diff HEAD`.**
დარწმუნდი, რომ commit-ში მხოლოდ ის მიდის, რაც გინდოდა — არც სატესტო
ხაზი, არც დროებითი ფაილი.

**Message = რა + რატომ.**
Diff აჩვენებს „რას" — message-ის ღირებულება „რატომ"-შია: რა იყო გატეხილი,
რა რისკი იყო, რა იცავს ახლა. სათაური ერთი ხაზი, ერთი ტიპი (feat/fix/
refactor — ურთიერთგამომრიცხავია: refactor ნიშნავს „ქცევა არ შეცვლილა").
ვინც message-ს ვერ წერს, მან ცვლილება ბოლომდე არ გაიგო.

## 9. Production Checklist — გაშვებამდე სავალდებულო

1. **HTTPS** — უამისოდ ყველაფერი ღია ტექსტით მოძრაობს; Secure cookie-ც
   ამას ელოდება.
2. **display_errors=Off, log_errors=On** — შეცდომები ლოგში, არა ეკრანზე.
3. **Secrets** — პაროლები/გასაღებები `.env`-ში, `.env` git-ში არასოდეს;
   production-ის პაროლები dev-ისგან განსხვავებული.
4. **Backup + აღდგენის ტესტი** — backup, რომელიც არასოდეს აღგიდგენია,
   backup არ არის. დაგეგმილი dump + პერიოდული აღდგენის ცდა.
5. **ლოგების კითხვის ჩვევა** — ლოგი, რომელსაც არავინ კითხულობს, არ არსებობს.
6. **Dev-ხელსაწყოები production-ში არ მიდის** — სატესტო ფაილები, seed
   მონაცემები, გახსნილი პორტები.
7. **უსაფრთხოების მინიმუმი დახურულია** — CSRF, სესიის გამაგრება,
   ownership, brute-force ლიმიტი, prepared statements ყველგან.

## 10. როგორ ფიქრობს სენიორი — მაქსიმები

- ჯერ უსაფრთხოება, მერე ყველაფერი დანარჩენი.
- არ ენდო არაფერს, რაც გარედან მოდის. არც საკუთარ მეხსიერებას — გადაამოწმე.
- ყოველ გადაწყვეტილებას აქვს „რატომ". თუ „რატომ"-ს ვერ ხსნი, გადაწყვეტილება არ არის შენი.
- დაცვა ისე ჩააშენე, რომ მისი დავიწყება შეუძლებელი იყოს.
- დაამტკიცე და არა „მგონია". ტესტი, ლოგი, header — მტკიცებულება.
- ივარაუდე, რომ სამი თვის მერე შენივე კოდს უცხო თვალით წაიკითხავ — იმისთვის დაწერე.
- პატარა, ხშირი, სუფთა ნაბიჯები ჯობია დიდ და იშვიათ ნახტომებს.
- როცა იღლები, აკეთე პატარა და კარგად შემოსაზღვრული საქმე — მაგრამ აკეთე.

მთავარი გაკვეთილი ამ ნაწილში: **ყველაფერი არჩევანია და ყველა არჩევანს
აქვს ფასი.** სენიორი განსხვავდება ჯუნიორისგან არა იმით, რომ „სწორ" გზას
ირჩევს, არამედ იმით, რომ იცის რა ალტერნატივები არსებობს, რა დათმო და
რატომ.

## 11. საიტის სრული ანატომია — ქვემოდან ზემოთ

ყველა საიტს, პატარასაც და დიდსაც, ეს ფენები აქვს:

1. **დომენი + DNS** ⏳ — სახელი (`example.com`) და ჩანაწერი, რომელი
   სერვერის IP-ზე მიდის ის.
2. **სერვერი/hosting** ⏳ — მანქანა, სადაც კოდი ცხოვრობს (VPS, cloud).
3. **Reverse proxy / ვებ-სერვერი** — nginx: იღებს მოთხოვნებს, აძლევს
   სტატიკურ ფაილებს, დანარჩენს PHP-ს გადასცემს. შენ ეს უკვე გაქვს Docker-ში.
4. **TLS/HTTPS** ⏳ — სერტიფიკატი (Let's Encrypt/Caddy), ტრაფიკის დაშიფვრა.
5. **Runtime** — php-fpm: პროცესები, რომლებიც შენს კოდს უშვებენ.
6. **აპლიკაციის კოდი** — შენი MVC. ამაზეა ნაწილი I.
7. **ბაზა** — MySQL. ცალკე პროცესი, ცალკე მოვლა (§13).
8. **სესიების საცავი** — ახლა ფაილები სერვერზე; დიდ სისტემებში Redis/ბაზა (§12).
9. **სტატიკური ფაილები** — CSS/JS/სურათები. nginx-ი აძლევს პირდაპირ,
   PHP-ს არ აწუხებს.
10. **Email-ის გაგზავნა** ⏳ — SMTP სერვისი (reset-ბმულებისთვის და ა.შ.).
    საკუთარი მეილ-სერვერის აწყობა ცალკე პროფესიაა — გამზადებული სერვისი აიღე.
11. **ლოგები** — ყოველი ფენა წერს თავისას: nginx access/error, PHP error, MySQL.
12. **Backup** ⏳ — ბაზის რეგულარული ასლი სხვა მანქანაზე.
13. **მონიტორინგი** ⏳ — ვინმემ უნდა შეამჩნიოს, რომ საიტი წაიქცა, სანამ
    მომხმარებელი შეამჩნევს (§15).

დაიმახსოვრე სურათად: **კოდი მთელი საიტის მხოლოდ ერთი ფენაა.** ამიტომ
„საიტის აწყობა" და „კოდის დაწერა" ერთი და იგივე არ არის.

## 12. არქიტექტურული გადაწყვეტილებები — არჩევანები და მათი ფასი

ყოველ პუნქტში: რა აირჩიე შენ (ან რას ირჩევენ ჩვეულებრივ), რა იყო
ალტერნატივა, როდის იცვლება პასუხი.

**Routing: კონვენცია თუ ცხრილი.**
შენი არჩევანი: კონვენცია — URL-დან იშლება `controller/action`.
პლუსი: ნული კონფიგურაცია, ახალი action ავტომატურად ჩანს.
მინუსი: URL-ები კოდის სტრუქტურას ამხელენ; ლამაზი მისამართები
(`/tasks/5/edit`) ძნელდება; ვერ აკონტროლებ, რომელი მეთოდია საჯარო.
ალტერნატივა ⏳: **route table** — ცხადი სია „ეს URL → ეს მეთოდი, მხოლოდ
GET/POST". ყველა framework ამას აკეთებს. როცა route-ების რაოდენობა
გაიზრდება ან API დაგჭირდება, ცხრილზე გადახვალ.

**დამოკიდებულებების აწყობა: ხელით factory თუ DI Container.**
შენი არჩევანი: ხელით — TaskWebsite ქმნის და აწვდის ყველაფერს.
პლუსი: ყველაფერი ხილულია, ჯადოქრობა ნულია, სწავლისთვის იდეალურია.
მინუსი: ყოველი ახალი კლასი = ხელით გაყვანილი „მავთულები"; იზრდება ერთი
დიდი „ღმერთი-ფაბრიკა".
ალტერნატივა ⏳: **DI Container** — ეუბნები „ინტერფეისს X პასუხობს კლასი Y"
და container თვითონ აწყობს ჯაჭვს. ფასი: ნაკლები ხილვადობა, მეტი მაგია.
წესი: ჯერ ხელით ისწავლე (შენ ეს გააკეთე — სწორია), container მერე.

**ბაზასთან მუშაობა: სამი სკოლა.**
შენი არჩევანი: **Table Gateway** — ერთი კლასი ერთ ცხრილზე, აბრუნებს მასივებს.
პლუსი: მარტივი, გამჭვირვალე, SQL-ს ხედავ და აკონტროლებ.
მინუსი: მონაცემი „უტვინო" მასივია — ქცევა (მაგ. „task ვადაგადაცილებულია?")
ცალკე ეხეტება.
ალტერნატივები ⏳: **Active Record** (Laravel Eloquent) — ობიექტი თვითონ
ინახავს თავს (`$task->save()`); სწრაფი წერა, მაგრამ model-ი ბაზაზეა
მიჯაჭვული და ტესტირება ჭირს. **Data Mapper** (Doctrine) — ობიექტები
ბაზის შესახებ არაფერს იციან, ცალკე mapper-ი გადააქვს; ყველაზე სუფთა და
ყველაზე რთული. წესი: მცირე პროექტს Gateway ჰყოფნის; ORM-ის ფასი მაშინ
იხდება, როცა domain-ლოგიკა მდიდარია.

**Templates: სუფთა PHP თუ ძრავი (Twig/Blade).** ⏳
შენი არჩევანი: PHP-ვე template-ებად.
პლუსი: არაფერი ახალი სასწავლი, სისწრაფე.
მინუსი: escape-ს **ხელით** აკეთებ ყოველ ჯერზე — ერთი დავიწყება = XSS.
ალტერნატივა: Twig/Blade escape-ს **default-ად** აკეთებს (choke point-ის
პრინციპი template-ებში!) და template-ში ბიზნეს-ლოგიკის ჩაწერას გიშლის.
დიდ პროექტში ან გუნდში — ძრავი თითქმის ყოველთვის იმარჯვებს.

**პასუხის მართვა: echo/header/exit თუ Response ობიექტი.** ⏳
ახლა controller-ები თვითონ იძახებენ header()-ს და exit()-ს.
პრობლემა: exit() კლავს ყველაფერს — ტესტში ასეთ მეთოდს ვერ გამოიძახებ;
პასუხს ვერ შეამოწმებ გაგზავნამდე.
ალტერნატივა: controller **აბრუნებს** Response ობიექტს (სტატუსი, headers,
ტანი), გაგზავნა კი ერთ ადგილას ხდება ბოლოში. ეს Phase 3-შია.

**ვალიდაცია: ფორმა-კლასები თუ წესების ენა.** ⏳
შენი არჩევანი: თითო ფორმას თავისი კლასი (RegisterValidation...).
პლუსი: ცხადი, ტესტირებადი. მინუსი: მეორდება საერთო ლოგიკა (სიგრძის
შემოწმება ყველგან თავიდან იწერება). ალტერნატივა: წესების ერთი ძრავი
(`'email' => 'required|email|max:254'` სტილში). როცა მეხუთე ვალიდატორს
დაწერ და გამეორება აგტკივდება — მაშინ.

**სესიების საცავი: ფაილები თუ ბაზა/Redis.** ⏳
ახლა: PHP-ს default — ფაილები სერვერზე. ერთ სერვერზე იდეალურია.
როცა ორი სერვერი გაჩნდება (load balancer), ფაილები აღარ მუშაობს —
მომხმარებელი ხან ერთ სერვერზე ხვდება, ხან მეორეზე, სესია კი პირველზეა.
მაშინ სესიები გადადის Redis-ში ან ბაზაში. ერთი სერვერი = ფაილები კარგია.

**Framework თუ vanilla — პატიოსანი პასუხი.**
Framework (Laravel, Symfony) გაძლევს: routing, DI container, ORM,
template ძრავი, ვალიდაცია, CSRF, migrations, ტესტების ინფრასტრუქტურა —
ყველაფერი, რასაც შენ ხელით აშენებ. მაშინ რატომ vanilla? იმიტომ, რომ
framework-ის მომხმარებელი, რომელსაც არ ესმის რას აკეთებს framework-ი
მის მაგივრად, უმწეოა როგორც კი რამე არასტანდარტული დასჭირდება. შენ
ახლა აშენებ იმ ცოდნას, რომელიც framework-ს გამჭვირვალეს გახდის.
სამსახურში Laravel-ს გამოიყენებ — და გეცოდინება, რას აკეთებს ის შიგნით.

## 13. ბაზის დიზაინი — რაც ცხრილების შექმნის მერე იწყება

**Foreign Keys და მთლიანობა.**
კავშირი ცხრილებს შორის ბაზის დონეზე ცხადდება (`FOREIGN KEY ... REFERENCES`).
რატომ: ბაზა თვითონ არ დაუშვებს „ობოლ" ჩანაწერს (task წაშლილი user-ით).
`ON DELETE` არჩევანია: CASCADE (გაჰყვეს), RESTRICT (აკრძალოს), SET NULL.
შენს schema-ში CASCADE დგას — user-ის წაშლისას მისი task-ები მიჰყვება.

**ინდექსები — ბაზის ყველაზე დიდი performance-ბერკეტი.** ⏳
ინდექსი ცხრილის „სარჩევია": უამისოდ ბაზა ყველა რიგს კითხულობს (full scan).
წესი: ინდექსი უნდა სვეტებს, რომლებზეც ხშირად ეძებ — WHERE, JOIN, ORDER BY.
შენი მაგალითი: `tasks.user_id`-ზე ინდექსი აუცილებელია — ყველა query
მასზე ფილტრავს. ფასი: ყოველი ინდექსი წერას ანელებს და ადგილს იკავებს —
„ინდექსი ყველაფერზე" ისეთივე შეცდომაა, როგორც „არცერთზე".

**Migrations — ბაზის ცვლილებებიც კოდია.** ⏳
შენი `01_schema.sql` მხოლოდ **ცარიელ** ბაზაზე ეშვება. როცა production-ში
მონაცემები იქნება და სვეტის დამატება მოგინდება — ხელით ALTER-ს ვერ
გაუშვებ ყველგან შეუმცდარად. Migration = დანომრილი ფაილი („005: დაამატე
last_attempt_at"), რომელიც ყველა გარემოში ერთნაირად და ერთხელ ეშვება.
ეს მოგცემს ბაზის ვერსიების ისტორიას git-ში, კოდის ისტორიის გვერდით.

**ტრანზაქციები — ან ყველაფერი, ან არაფერი.** ⏳
როცა ერთი ოპერაცია რამდენიმე ჩანაწერს ცვლის (მაგ. შეკვეთა + საწყობის
შემცირება), ისინი ერთ ტრანზაქციაში ეხვევა: შუაში ჩავარდნისას ბაზა
საწყის მდგომარეობას უბრუნდება. უამისოდ ნახევრად-შესრულებული ოპერაცია
ბაზას წინააღმდეგობრივ მდგომარეობაში ტოვებს.

**N+1 პრობლემა — ORM-ების კლასიკური ხაფანგი.** ⏳
სიაში 100 task-ია და თითოს user-ის სახელი უნდა → 1 query სიისთვის +
100 query სახელებისთვის = 101. სწორია: 1 query JOIN-ით ან IN-ით.
Gateway-სტილში ამას თვალით ხედავ; ORM-ში — ჩუმად გემართება, ამიტომ
ORM-თან ლოგში query-ების რაოდენობას უყურებ.

**კოდირება და რეჟიმი.**
`utf8mb4` ყველგან (utf8-ად შენიღბული ძველი MySQL-კოდირება emoji-ს ჭრის);
strict mode ჩართული — ბაზამ **შეცდომა** უნდა ისროლოს ზედმეტ სიგრძეზე
და არა ჩუმად მოჭრას მონაცემი.

## 14. Performance — წესები, თანმიმდევრობით

**წესი 0: ჯერ გაზომე, მერე ააჩქარე.**
ოპტიმიზაცია გაზომვის გარეშე ვარაუდია. ლოგი, slow query log, browser-ის
Network tab — ჯერ იპოვე სად იკარგება დრო, მერე შეეხე კოდს.
Premature optimization ართულებს კოდს იქ, სადაც პრობლემა არც იყო.

დამხმარეების რიგი, იაფიდან ძვირისკენ:
1. **opcache** ⏳ — PHP-ს კომპილირებული კოდი მეხსიერებაში რჩება.
   ერთი ჩართვა php.ini-ში, production-ში სავალდებულო.
2. **ინდექსები** ⏳ — §13. ყველაზე დიდი ეფექტი ყველაზე იაფად.
3. **Pagination** — უსასრულო სია ნელდება და მეხსიერებას ჭამს.
   `findAllTasks` უlimit-ოა — ეს Phase 7-შია შენთან.
4. **ქეშირება** ⏳ — ჯერ browser-ის ქეში სტატიკაზე (headers), მერე
   აპლიკაციის ქეში (იშვიათად ცვალებადი, ძვირი query-ების შედეგი).
   ქეშის ფასი: invalidation — „როდის აღარ არის ეს პასუხი სწორი?" —
   პროგრამირების ორი ურთულესი პრობლემიდან ერთია.
5. **CDN/asset-ოპტიმიზაცია** ⏳ — მაშინ, როცა რეალური ტრაფიკი გაქვს.

## 15. ოპერაციები — საიტი გაშვების მერე იწყება

**Deployment — კოდის გატანა წესით.** ⏳
Dev-ში ფაილები volume-ით ედება; production-ში კოდი **image-შია ჩაწყობილი**
(COPY) — გაშვებული კონტეინერი უცვლელია, ახალი ვერსია = ახალი image.
მიზანი: ნებისმიერ მომენტში ზუსტად იცი, რომელი კოდი მუშაობს, და უკან
დაბრუნება ერთი ბრძანებაა.

**მონიტორინგი — შეიტყვე მომხმარებელზე ადრე.** ⏳
მინიმუმი: გარე uptime-შემოწმება (ყოველ წუთს უკითხავს საიტს „ცოცხალი
ხარ?") + შეტყობინება შენთან. შემდეგი დონე: error-ების რაოდენობის ზრდაზე
სიგნალი. ლოგი, რომელსაც კვირაში ერთხელ კითხულობ, მონიტორინგი არ არის.

**Dependencies-ის მოვლა.** ⏳
`composer audit` — ცნობილი ხვრელების შემოწმება შენს პაკეტებში.
PHP-ს და პაკეტების ვერსიები ძველდება; განახლება გეგმიური საქმეა და არა
საგანგებო. მიტოვებული dependency = მომავალი უსაფრთხოების ინციდენტი.

**Rate limiting ყველგან, არა მხოლოდ login-ზე.** ⏳
რეგისტრაცია, პაროლის აღდგენა, ძებნა — ყველაფერი, რაც ძვირია ან
ბოროტად გამოიყენება, ლიმიტდება (X მოთხოვნა Y დროში ერთი IP/ანგარიშიდან).

## 16. მომხმარებლის მხარე — რასაც კოდის მიღმა ივიწყებენ

**Accessibility (a11y).** ⏳
საიტი კლავიატურით უნდა იმართებოდეს და screen reader-ს ესმოდეს.
წესები: სემანტიკური ელემენტები (`<button>` და არა click- იანი `<div>`;
შენს checkbox-ებს ეს პრობლემა აქვთ — Phase 7), label ყველა input-ს,
alt ყველა სურათს, კონტრასტული ფერები.

**Mobile-first.** ⏳
ტრაფიკის ნახევარზე მეტი ტელეფონია. განლაგება ჯერ ვიწრო ეკრანზე
მუშაობდეს, მერე გაფართოვდეს.

**SEO-ს საფუძველი = ტექნიკური სისწორე.** ⏳
სემანტიკური HTML (h1 ერთია, სათაურები იერარქიით), სწორი სტატუს კოდები
(404 იყოს 404 და არა 200 ცარიელი გვერდით), title/description ყველა
გვერდს, ადამიანური URL-ები.

**პერსონალური მონაცემები — შეაგროვე მინიმუმი.** ⏳
წესი: რაც არ გაქვს, ვერ მოგპარავენ. ინახავ მხოლოდ იმას, რაც საჭიროა;
მომხმარებელს შეუძლია ანგარიშის წაშლა და მონაცემები მართლა იშლება;
პაროლები ჰეშად, ლოგებში პერსონალური მონაცემი არ იწერება.
